## 1. Architecture Overview

SnapSquad V1 uses a **"Blind Coordinator"** architecture. The server **never** sees raw photos or raw face embeddings. All AI/ML processing happens on the mobile device. The server's only intelligence is HDBSCAN clustering — pure math, zero ML models.

```
┌─────────────────────────────────────────────────────────────────────┐
│                         MOBILE DEVICE                               │
│                                                                     │
│   ┌──────────┐    ┌──────────────┐    ┌───────────────────────┐    │
│   │  Camera/  │───►│ SCRFD-2.5GF  │───►│   MobileFaceNet       │    │
│   │  Gallery  │    │  (TFLite)    │    │   (TFLite, 512-d)     │    │
```

### What the Server Knows vs What It Doesn't

| Server CAN See | Server CANNOT See |
|---|---|
| Protected embeddings (random-looking 512-d vectors) | Raw face embeddings |
| Encrypted photo blobs | Actual photos |
| Cluster labels (face X belongs to group 3) | Who the person is |
| Album metadata (name, photo count) | Photo contents |
| User accounts (email, hashed password) | Plaintext passwords |

## 3. Security & Privacy Architecture

### Three Layers of Protection

```
Layer 1: FEATURE SUBTRACTION (on device)
├── Raw embedding ∈ ℝ⁵¹² (128-byte face fingerprint)
├── Private template ∈ ℝ⁵¹² (stored in Android Keystore via expo-secure-store)
├── protected = raw - template
├── Server receives only protected embedding
├── Inversion is computationally infeasible without template
└── Key insight: cosine distances are PRESERVED after subtraction
    so HDBSCAN produces identical clusters on protected embeddings

Layer 2: AES-256 ENCRYPTION (on device)
├── Photo encrypted before leaving device
├── Server stores opaque encrypted blobs
├── Server cannot view, read, or analyze photos
└── Decryption happens only on authorized device

Layer 3: STANDARD AUTH
├── Passwords: bcrypt-hashed (never stored in plaintext)
├── Sessions: JWT tokens with configurable expiry
├── Album ownership: every endpoint checks user_id matches
└── Rate limiting: TODO in V2 (nginx/API gateway layer)
```

### Why Feature Subtraction Works (CVPR 2024)

```
Given:  raw_A, raw_B   (two embeddings of the same person)
        template        (user's private random vector)

Protected:  prot_A = raw_A - template
            prot_B = raw_B - template

Cosine similarity:
    cos(prot_A, prot_B) ≈ cos(raw_A, raw_B)

    This holds because subtracting the same vector preserves
    the relative angular relationships between embeddings.

    HDBSCAN uses cosine distance → same clusters either way.
```

## 4. Tech Stack

| Component | Technology | Why |
|---|---|---|
| Web Framework | FastAPI 0.115+ | Async-capable, auto-docs, type safety via Pydantic |
| Database | PostgreSQL + SQLAlchemy 2.0 | Relational integrity, pgvector future, ACID |
| DB Driver | psycopg 3 (binary) | Modern async-capable PostgreSQL driver |
| Auth | python-jose (JWT) + passlib (bcrypt) | Industry standard token + password hashing |
| Job Queue | Redis + RQ | Lightweight, Python-native background jobs |
| Clustering | HDBSCAN + scikit-learn + numpy | Density-based clustering, no arbitrary k |
| Blob Storage | Cloudinary (prod) / Filesystem (dev) | 25GB free tier, CDN delivery |
| Config | pydantic-settings | Type-safe .env loading with validation |

### What Was Deliberately NOT Included

| Removed | Reason |
|---|---|
| InsightFace | ML detection/embedding runs on device, not server |
| ONNX Runtime | No ML inference on server |
| OpenCV | No image processing on server (photos are encrypted) |
| TensorFlow/PyTorch | Zero ML frameworks needed |

## 8. HDBSCAN Clustering Pipeline

The clustering algorithm (`cluster_faces_v21`) is the **heart** of the backend — and the only real computation it does. It's a 10-stage pipeline tuned from extensive notebook experiments.

### Pipeline Stages

```
INPUT: N protected face embeddings (512-d each)
  │
  ▼
┌─────────────────────────────────────────────┐
│ Stage 1: COSINE DISTANCE MATRIX             │
│ Pairwise 1 - cos_sim between all faces      │
│ Cast to float64 for HDBSCAN Cython          │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 2: HDBSCAN CLUSTERING                 │
│ metric=precomputed, min_cluster_size=2,     │
│ min_samples=1, cluster_selection_epsilon=0.3│
│ → initial cluster labels                    │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 3: POST-VALIDATION                    │
│ For each cluster: check avg_sim ≥ 0.30,     │
│ min_link ≥ 0.25. Eject weak faces.          │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 4: MERGE PASS 1                       │
│ Merge clusters if centroid_sim > 0.55       │
│ AND cross_min_sim ≥ 0.20                    │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 5: MERGE PASS 2                       │
│ More aggressive merge (different thresholds)│
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 6: RE-VALIDATION                      │
│ Re-check all clusters after merges          │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 7: COHERENCE SPLIT                    │
│ Split incoherent clusters (mixed people)    │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 8: RESCUE PASS 1                      │
│ Re-assign outliers to nearby clusters       │
│ if similarity > rescue threshold            │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 9: ADAPTIVE OUTLIER EJECTION          │
│ Remove borderline faces from rescue results │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ Stage 10: RESCUE PASS 2                     │
│ Final rescue attempt for remaining outliers │
└──────────────────────┬──────────────────────┘
                       ▼
OUTPUT: labels[] (cluster ID per face, -1 = outlier)
        summary  (n_faces, n_clusters, n_clustered, stats per stage)
```

### Pipeline Configuration (Tuned Thresholds)

```
HDBSCAN_CONFIG:
    min_cluster_size     = 2
    min_samples          = 1
    cluster_selection_epsilon = 0.3

POST_VALIDATION:
    avg_threshold        = 0.30
    min_link_threshold   = 0.25

MERGE_PASS1:
    centroid_threshold   = 0.55
    cross_min_threshold  = 0.20

MERGE_PASS2:
    (separate thresholds for aggressive merging)

RESCUE_PASS1 / RESCUE_PASS2:
    (thresholds for re-assigning outliers)

    (adaptive ejection parameters)
```

## 9. Storage Layer

### Dual Backend Architecture

```
┌─────────────────────────────────────────────────┐
│           get_storage_backend()                   │
│  (factory function in cloudinary_store.py)        │
│                                                   │
│  if settings.cloudinary_configured:               │
│      return CloudinaryStore()   ◄── Production    │
│  else:                                            │
│      return LocalStore()        ◄── Development   │
└─────────────────────────────────────────────────┘
```

Both backends implement the same interface:
- `save_encrypted_blob(album_id, photo_id, data) → url`
- `read_encrypted_blob(album_id, photo_id) → bytes`
- `delete_encrypted_blob(album_id, photo_id)`

**Important**: The server stores encrypted blobs as opaque binary data. It has no ability to decrypt or view the photos. Only the mobile app (which holds the AES key) can decrypt them.

## 10. Background Job Queue

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│  FastAPI App  │  enqueue │    Redis     │  dequeue │  RQ Worker   │
│              │─────────►│   Queue      │─────────►│              │
│ POST /process│          │  "pipeline"  │          │ process_     │
│              │          │              │          │ album_job()  │
└──────────────┘          └──────────────┘          └──────┬───────┘
                                                           │
                                                           ▼
                                                    ┌──────────────┐
                                                    │ 1. Load       │
                                                    │    embeddings │
                                                    │    from DB    │
                                                    │               │
                                                    │ 2. Run 10-    │
                                                    │    stage      │
                                                    │    HDBSCAN    │
                                                    │               │
                                                    │ 3. Write      │
                                                    │    labels     │
                                                    │    back to DB │
                                                    │               │
                                                    │ 4. Create     │
                                                    │    Cluster    │
                                                    │    records    │
                                                    │               │
                                                    │ 5. Update     │
                                                    │    album →    │
                                                    │    "complete" │
                                                    └──────────────┘
```

**Note on Windows**: RQ requires Unix `fork()` which doesn't exist on Windows. The import is lazy (`get_queue()`) so the FastAPI server starts fine on Windows for development. The worker process must run on Linux (or WSL) in production.

## 13. Key Design Decisions

| Decision | Rationale |
|---|---|
| **No ML on server** | Architecture docs mandate server blindness. Detection + embedding on device via TFLite. |
| **Feature subtraction over homomorphic encryption** | 6 approaches evaluated (see Doc 4). Feature subtraction is fastest, simplest, and preserves cosine distance for clustering. |
| **HDBSCAN over K-Means** | HDBSCAN doesn't require specifying k (number of people is unknown). Density-based → handles variable group sizes. |
| **RQ over Celery** | Lighter weight. Single queue, Python-native. Celery is overkill for V1 single-machine deployment. |
| **Lazy RQ imports** | RQ uses `fork()` which doesn't exist on Windows. Lazy init lets the server start on Windows for development. |
| **Cloudinary + LocalStore abstraction** | `get_storage_backend()` factory returns the right backend based on config. Swap without code changes. |
| **JSON text for embeddings (V1)** | Avoids pgvector dependency for V1. Easy migration path via ALTER TABLE later. |
| **Audit log for edits** | ClusterEdit table logs every rename/merge/eject. Enables undo in V2 and debugging. |
| **Album ownership on every endpoint** | Every protected endpoint checks `album.user_id == current_user_id`. No cross-user data leaks. |
| **UUID primary keys** | No predictable auto-increment IDs. Prevents enumeration attacks. |

## 14. How to Run

### Prerequisites

1. **PostgreSQL** installed and running (port 5432)
2. **Redis** installed and running (port 6379)
3. **Python 3.12+**

### Setup

```bash
# 1. Create the database
psql -U postgres -c "CREATE DATABASE snapsquad;"

# 2. Create .env from template
cd backend
cp .env.example .env
# Edit .env with your DATABASE_URL, JWT_SECRET, etc.

# 3. Install dependencies
pip install -r requirements.txt

# 4. Start the API server (auto-creates all 9 tables)
uvicorn app.main:app --reload --port 8000

# 5. Start the worker (separate terminal, Linux/WSL only)
python scripts/start_worker.py
```
